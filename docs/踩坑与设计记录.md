# 踩坑与设计记录

项目跑通以后回头看，最费时间的并不是把五个节点连起来，而是处理节点之间怎么传数据、
Critic 打回后怎么收场，以及如何保证报告里的引用没有在几轮生成中乱掉。这篇主要记录这些
和 Agent 流程直接相关的问题。模型下载和本地环境的坑放在最后简单带过。

## 先说明这里的“多智能体”是什么

项目里的 Planner、Retriever、Analyst、Critic、Writer 共用同一个 LLM 端点，并不是五个
模型各自运行。Retriever 甚至不调用 LLM，只负责向量检索和重排。更准确地说，这是一条用
LangGraph 编排的多角色 Agent 工作流：每个节点职责不同，通过共享 state 交接结果。

我保留这种拆分，主要是因为它比一个大 prompt 更容易控制：Planner 的拆题结果可以单独看，
Retriever 可以替换或重跑，Critic 也能通过条件边决定是否回到检索。哪个环节出问题，在
trace 里基本能直接定位。

当前流程是：

```text
START → Planner → Retriever → Analyst → Critic ──pass/到达上限──→ Writer → END
                    ↑                         │
                    └──────── revise ─────────┘
```

## State 不能只当聊天记录用

节点之间传的是 `ReportState`，里面分别保存子问题、检索证据、引用表、分析、Critic 意见和
最终报告。这里没有把所有内容都塞进一段不断增长的对话历史，而是让每个节点只读自己需要的
字段，再返回本轮修改的部分。

有个细节很重要：第二轮检索得到的 `evidence` 和 `citations` 会整体替换第一轮结果，不做
追加。如果新旧证据混在一起，Analyst 很难知道哪些是 Critic 打回前的旧结果，引用编号也会
越来越难维护。LangGraph 默认的后写覆盖正好符合这个需求，所以这两个字段没有使用
`operator.add` 一类的 reducer。

`revise_count` 也放在 state 中，而不是存在某个节点的局部变量里。这样无论是条件路由、测试
还是流式跟踪，都能看到当前已经审了几轮。

## Critic 最先遇到的问题是停不下来

最初的逻辑是：Critic 不满意就回 Retriever，直到它给出 `pass`。真实跑一次就会发现这不行。
如果语料里根本没有自由现金流、历史估值分位数之类的数据，重新检索不会凭空把它们变出来，
Critic 却会继续要求补充，流程就一直卡在 Retriever、Analyst、Critic 之间。

现在用了两层限制：

1. Critic 每运行一次就增加 `revise_count`。配置 `max_revise=2` 时，第二次审查结束后无论
   verdict 是什么，都进入 Writer。
2. Critic 只输出 `verdict` 和 `notes`，不直接决定下一个节点。真正的跳转统一放在
   `route_after_critic` 中。修改轮次上限或退出条件时，不需要动 Critic 的 prompt。

还有一种情况来自 LLM 输出本身。Critic 虽然被要求返回 JSON，但偶尔还是会带解释文字，或者
输出约定之外的 verdict。当前实现选择 fail open：JSON 解析失败或 verdict 非法时按 `pass`
处理。对这个 demo 来说，产出一份不够完整的草稿比工作流永远不结束更可接受。生产环境可以
在这里增加一次结构化重试，或者把“解析失败”和“审核通过”拆成两个状态。

## 打回不能只是把原流程再跑一遍

如果 revise 之后仍然用原来的子问题重新检索，通常只会拿回相同结果。现在 Retriever 会读取
`critique.notes`，把它追加为一条补充 query：

```text
针对以下不足补充证据：{Critic 的意见}
```

例如第一轮漏了经营现金流，Critic 的 notes 会让第二轮多查一条相关证据。原来的子问题仍然
保留，因此第二轮既不会丢掉基础分析，也能补查具体缺口。

这里更准确的叫法是“Self-RAG 式反思重检索”。它借用了生成后检查、再检索的思路，但没有
实现论文里的反思 token 或训练过程，不是完整的 Self-RAG。当前 notes 还是一句自由文本，
下一步如果继续做，可以让 Critic 输出结构化的 `missing_facts`，Retriever 再逐项生成 query，
会比把整句意见直接拿去检索稳定。

## 引用约束不能全靠 prompt

Analyst 的 prompt 要求每条结论附 `[n]`，但只写这句规则并不能保证编号始终正确。这个项目
另外维护了一张全局引用表：Retriever 对来源去重并分配编号，Analyst 只能看到这张表中的编号，
Writer 最后的“引用来源”则由代码直接渲染，不交给 LLM 生成。

这样至少解决了两个问题：同一来源不会在不同子问题里拿到多个编号，文末的 `[3]` 也不会在
Writer 改写正文时变成另一个来源。Critic 再从语义上检查结论是否真的得到引用内容支持。

目前仍有一个缺口：正文里的编号还没有做程序化校验。LLM 理论上仍可能写出不存在的 `[99]`，
或者引用存在但内容并不支持结论。后者需要 Critic 判断，前者其实可以在 Writer 之后用正则和
引用表直接检查。与其继续加 prompt，这类确定性约束更适合交给代码。

## 每个节点都要有明确的输出协议

Planner 应该返回 JSON 数组，Critic 应该返回固定字段的 JSON，Analyst 和 Writer 才输出
Markdown。几个节点如果都自由发挥，后面的解析和路由很快就会变成猜测。

即便规定了格式，仍然要准备降级路径：

- Planner 解析失败时使用预设的四个分析维度；
- Critic 用 `parse_json` 兼容代码块和前后解释，仍失败则走安全默认值；
- Writer 不负责生成引用列表，只负责整理已经带引用的分析；
- 没有 API key 时使用离线 stub，让图、路由和引用逻辑仍然可以测试。

离线 stub 不能用来判断报告质量，它的作用只是把 Agent 的控制流和外部模型调用分开。CI 不需要
访问付费 API，也能确认 revise 回路确实触发、补充 query 确实加入、最终能走到 END。

## 调 Agent 流程时，trace 比最终报告更有用

只看最后的研报，很难判断问题发生在哪一层。报告缺了一项数据，可能是 Planner 没拆出来，
也可能是 Retriever 没召回、Analyst 漏写，或者 Critic 没发现。

`scripts/trace_run.py` 使用 LangGraph 的 `.stream()` 逐节点打印更新内容。我通常会看这几项：

- Planner 实际拆出了哪些子问题；
- 每条 query 命中了几段，引用编号是否连续；
- Critic 给了什么 verdict，notes 是否足够具体；
- revise 后是否真的多出一条补充 query；
- Writer 是否保留了 Analyst 的引用编号。

对应的测试也尽量测控制流，而不去断言 LLM 应该写出某句话。目前覆盖了图能否编译、反思回路
是否触发、补充检索是否加入、引用表是否去重且连续，以及最终报告是否包含引用列表。

## Agent 的上限往往卡在工具，不在 prompt

第一次换成真实 embedding 后，问“贵州茅台的 ROE”却召回了别的公司。原因很简单：chunk 里
只有 `600519`，用户问题里只有公司名，embedding 模型并不知道两者是同一个标的。给 chunk
补上 `贵州茅台(600519)` 后重建索引，相似度从约 0.5 提升到约 0.8。

这件事也影响了我对 Critic 的理解。Critic 可以发现证据不够，却无法修复一个信息表示错误的
检索库。Agent 的反馈回路不是兜底一切的魔法；底层工具返回什么，它就只能在这些结果上继续
推理。

评测集也踩过类似的坑。第一版 8 道题都问同一家公司，而这家公司刚好只有 5 个 chunk，
`k_final` 也是 5。无论是否重排，送给 LLM 的都是同样内容，三组配置当然没有差异。后来换成
15 道题，加入跨公司比较和不带公司名的筛选题，才真正测到检索与重排的区别。

通用 bge-reranker 在这组财务文本上反而降低了结果：faithfulness 从 0.800 降到 0.644。
用领域 query 和难负样本微调后回到 0.800，context precision 从 0.280 提升到 0.347。这个结果
更适合说明“组件需要在实际链路里验证”，不能据此推断通用模型在所有金融任务上都不好。

## 其它工程问题

这些问题也花了不少时间，但和 Agent 设计关系不大，只保留结论：

- venv 移动目录后，入口脚本里的绝对 shebang 会失效；调用依赖工具时用 `python -m ...`
  更稳妥。
- transformers、FlagEmbedding 和 torch 在 macOS 上要成组锁版本，单独升级其中一个容易出现
  API 不兼容甚至段错误。
- faiss-cpu 和 torch 会加载两份 OpenMP 运行时；当前在导入 faiss 前设置
  `KMP_DUPLICATE_LIB_OK=TRUE` 和 `OMP_NUM_THREADS=1`。
- akshare 请求可能长期不返回。批量抓取时按股票拆成子进程，并给每个子进程设置硬超时，避免
  一只股票卡住整批任务。
- 弱网下载大模型时不能只看文件大小。`scripts/robust_download.py` 强制检查 HTTP 206，并在
  下载完成后校验 SHA256。

## 实验结果的口径

- reranker 微调：201 条训练数据、33 条验证数据；dev acc@1 从 0.879 提升到 1.0。端到端
  评测只有 15 道题，因此这里主要看趋势，不把小数点后的差异当成普遍结论。
- SFT：用 201 条金融 RAG 问答对 Qwen2.5-7B 做 LoRA。15 道同题评测中，学生模型的
  answer correctness 为 0.800，教师模型为 0.793；这个结果说明小模型在当前数据分布上可用，
  不能简单外推成整体能力相当。
