# 踩坑与设计记录

记录这个项目开发过程中真正卡住过我的问题，和几个关键设计决策的来龙去脉。
本地环境是 MacBook Air M2 / 16GB，网络条件比较差（热点 + VPN），云端训练用的 AutoDL 上的 RTX 4090。

## 一、多智能体系统的设计问题

### Critic 反思回路的死循环

系统里 Critic 判 revise 会把流程打回 Retriever 重新检索。最初的想法很朴素：质量不行就重做，直到 Critic 满意为止。

真跑起来才发现问题：**如果语料里根本没有 Critic 想要的数据，它永远不会满意**。有一次跑贵州茅台，Critic 第一轮说缺自由现金流和 PE 历史分位数，重新检索之后第二轮又说缺预收账款——这些数据在语料里就不存在，再检索一百次也补不回来，流程会在 retriever → analyst → critic 之间无限转圈。

最后的解法分两层：

1. state 里加 `revise_count` 计数器，每过一次 Critic 加一。路由函数里写死：`verdict == "pass"` 或 `revise_count >= max_revise`（配置为 2）都放行去 Writer。上面茅台那次就是第二轮撞上限被强制出稿的。
2. 判断和路由分离。Critic 只输出 verdict + notes，它不知道也不决定流程去哪，真正的跳转在 graph 的条件边 `route_after_critic` 里。防死循环的逻辑集中在一个函数，改上限、改条件都不用碰 Critic 本身。

还有一个更隐蔽的死循环入口：LLM 输出不可控。Critic 被要求只输出 JSON，但它偶尔会夹带解释文字，或者把 verdict 写成"需要修改"这种约定之外的值。如果解析失败就重试、或者默认按 revise 处理，一样会卡死。所以这里的策略是**拿不准就放行**——parse_json 解析失败默认 pass，verdict 非法也强制改成 pass。宁可放出一份不够完美的报告，也不能让流程挂死在那里。

### 为什么要有 Critic

最早的链路是 Planner → Retriever → Analyst → Writer 一遍过。跑几次就能看出毛病：Analyst 的输出质量不稳定，有时结论没有证据支撑（典型的幻觉），有时干脆漏掉一个子问题。一遍过的流水线对此毫无办法。

加 Critic 本质是给系统加一个反馈回路。它检查三件事：有没有 [编号] 引用撑不住的结论、有没有漏答的子问题、数据前后有没有矛盾。更重要的是它的 notes 会被 Retriever 拼成一条补充检索 query（"针对以下不足补充证据：……"），第二轮检索是带着具体问题去的，不是盲目重跑一遍——这个思路来自 Self-RAG。

效果在 trace 里能直接看到：第一轮 Critic 打回后，第二轮 Retriever 的 query 从 5 条变成 6 条（多的那条就是 Critic 的意见），分析里补上了经营现金流的具体数字。

其它几个角色的取舍顺带记一下：

- Planner 单独存在，是因为把"分析茅台"拆成 4~5 个子问题后可以逐题检索，召回比拿一整句话去查准得多；
- Writer 单独存在有个不显眼的原因：最终报告的引用列表是代码从 citations 表直接渲染的，不经过 LLM。这样正文里的 [3] 和文末列表的 [3] 永远指向同一个来源，LLM 没机会篡改或编造引用；
- Retriever 是唯一不调 LLM 的节点。检索这种事向量库又快又准还便宜，LLM 只留给它擅长的理解和生成。

## 二、Python 环境

**conda base 把 venv 挡了。** 终端提示符明明显示 `(.venv)`，`python main.py` 却报 langgraph 找不到。原因是提示符实际是 `(.venv) (base)` 双前缀，conda 的路径在 PATH 里排在 venv 前面，`python` 解析到的是 anaconda 的解释器。不确定的时候先跑一句 `python -c "import sys; print(sys.executable)"` 验明正身，能省很多瞎猜的时间。根治办法是 `conda config --set auto_activate_base false`。

**venv 里的命令行工具 shebang 失效。** `.venv/bin/pip`、`.venv/bin/hf` 报 bad interpreter，指向一个不存在的旧路径——venv 建好之后项目目录改过名，这些入口脚本的 shebang 是写死的绝对路径。教训是 venv 不能移动或改名；补救办法是一律用 `python -m pip` 这种形式调用，不依赖入口脚本。

## 三、弱网环境下模型权重的下载

这部分坑最多，前后折腾了一晚上，按时间线记：

1. `snapshot_download` 走 hf-mirror 卡在 0%，几分钟零字节。查下来是新版 huggingface_hub 默认走 Xet 传输协议，镜像站不支持。`HF_HUB_DISABLE_XET=1` 可以绕过，或者干脆 curl 直连 `resolve/main/<file>` 的文件 URL。

2. 换 curl 断点续传（`-C -`）后出现了更阴间的问题：下载进度不升反降（88MB 掉回 74MB），最后文件大小对了但加载时段错误。原因是部分 CDN 节点会忽略 Range 请求直接返回 200 全量，curl 就截断重写整个文件；限流时的 HTML 错误页也可能被当成文件内容写进去。**文件大小对不等于文件是好的。**

3. 于是自己写了个分块下载器（scripts/robust_download.py）：8MB 一块发 Range 请求，逐块落盘重试，服务器不返回 206 就直接作废换源，进度只增不减。下载完成后拿官方 SHA256 校验——HF 的 LFS pointer 和 ModelScope 的 API 都能查到哈希。另外双镜像源轮换续传的前提是两边文件字节级一致，混用之前先比对两站的 SHA256。

4. 分块下载器也卡过一次：文件几分钟不动但进程活着。这次是 requests 的坑——它的 timeout 是**字节间隔超时**，服务器每 59 秒滴一个字节，连接就永远不超时，一个 8MB 的块理论上能吊死几个小时。改成 stream 流式读取加整块 90 秒的墙钟死线，到点砍连接；砍掉时已收到的半块是有效的前缀字节，照样落盘不浪费。

5. 速度层面的经验：hf-mirror 被反复请求后触发限流（新连接也只给十几 KB/s），这种时候优化重连没用，直接换 ModelScope（阿里的 CDN，当时实测 880KB/s）。另外 VPN 全局模式会把国内源也绕道海外，SSL 频繁重置，下国内源之前先把 VPN 关掉或切规则分流。

6. 一个纯 Python 的坑：macOS 上 python.org 安装的 Python 没跑过 Install Certificates.command 的话，urllib 访问任何 HTTPS 都报证书错误（curl 正常，因为它用系统钥匙串）。换 requests 就好，它自带 certifi。

## 四、macOS 上的深度学习栈兼容性

三个段错误连环撞上，都是 exit 139，Python 层没有任何异常信息：

1. transformers 5.13 的新版多线程权重加载器在 macOS + torch 2.13 上加载模型直接 Segmentation fault。降级到 4.51.3 解决。定位手段：`PYTHONFAULTHANDLER=1` 加 `python -u`，能拿到 C 层崩溃时的 Python 调用栈。

2. 降完 transformers 又发现新版 FlagEmbedding 按 transformers 5 的 API 传参（`dtype=`），4.x 只认 `torch_dtype=`。配套降到 FlagEmbedding 1.2.11 才配平。深度学习的依赖要**成组锁版本**，单独动一个很容易连环崩。

3. 最顽固的一个：单独用 FlagModel 编码没事，单独用 faiss 没事，两个在同一个进程里（建索引恰好需要）就段错误。原因是 faiss-cpu 和 torch 各静态链接了一份 OpenMP 运行时，同进程加载两份是未定义行为。修复是在 import faiss 之前设 `KMP_DUPLICATE_LIB_OK=TRUE` 和 `OMP_NUM_THREADS=1`，两个都要，已经固化在 src/rag/index.py 里。

## 五、数据与检索

**akshare 接口没有超时，会永久挂起。** 批量拉 20 支股票，卡在第 10 支一动不动，也不报错。它底层的请求不设 timeout，接口抽风就无限等。改成每支股票开独立子进程跑，`subprocess.run(..., timeout=90)` 硬超时，拉不动的直接跳过（最后 17/20 成功，够用）。批处理任务的单元之间要做故障隔离，外部接口永远按"它会挂起"来设防。

**chunk 文本里只有股票代码，检索指鹿为马。** 换上真实 embedding 后第一次测试，问"贵州茅台的 ROE"命中的是长江电力和中国中免，问"宁德时代"前三名全是别家公司。盯着结果看了一会儿才反应过来：chunk 文本是 `600519 截至...` 这种裸代码开头，而人提问用的是公司名，通用 embedding 模型不可能知道贵州茅台等于 600519。给 loader 加了个代码到名称的映射，chunk 改成 `贵州茅台(600519) 截至...`，重建索引后三类提问全部命中正确公司，相似度从 0.5 涨到 0.8。这是整个项目里投入产出比最高的一次修复——**检索效果差先查喂给模型的数据，再怀疑模型和参数**。

**评测集没有区分度，三组配置分数完全一样。** 第一版评测集 8 道题全问茅台，而语料里茅台恰好只有 5 个 chunk，k_final 也是 5——召回阶段就把这 5 条全捞出来了，排序排得再好喂给 LLM 的都是同样 5 条，重排器根本没有发挥空间。后来把评测集重做成 15 题三个梯度：多公司单点、跨公司对比（top-5 必须同时装下两家公司的正确 chunk）、无公司名筛选（问题里没有公司名可供匹配，纯靠语义排序），差异立刻出来了。反过来说，"各配置分数相同"本身也是有效信号，说明瓶颈不在被测的那个组件上。

顺带记一下评测出来的一个反直觉结论：通用 bge-reranker-base 在这个语料上是**负收益**（faithfulness 0.80 掉到 0.64），它没见过"表格式中文财务数字文本"，重排反而把对的排下去了；用 LLM 造 query 加难负样本微调之后才收复反超。"上了重排就一定更好"并不成立，垂直领域要实测。

## 六、工程习惯

**管道会吞掉进度和错误。** 后台任务显示"完成"但日志是空的、明明失败了 exit code 却是 0——因为 `python ... | tail` 这种写法里 stdout 有缓冲（段错误时缓冲区内容直接丢失），管道的退出码又取决于最后一环。长任务的正确姿势：`python -u` 关缓冲、输出重定向到日志文件而不是经过管道、结尾显式 `echo exit=$?`。

**长任务必须有心跳。** 下载和训练这类任务，"沉默"和"正常运行"从外面看起来一模一样。这次的做法是单独起一个监控循环，每分钟看一次文件大小增量或日志尾行，停滞就报警。SSH 到云端跑训练还有一个必须的动作：命令用 nohup（或 tmux）脱离会话跑——AutoDL 的网关会掐长连接，第一次装 torch 就是装到一半连接被断，白等了十分钟。
